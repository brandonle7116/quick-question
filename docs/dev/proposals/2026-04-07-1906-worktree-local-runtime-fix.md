## Problem

`/qq:execute` and `/qq:go` skills (since v1.16.21) tell agents to enter a worktree via Claude Code's built-in `EnterWorktree` tool, then call `qq-worktree.py seed-runtime-cache` to seed `Library/`. This procedure leaves the new worktree **incomplete** in two ways:

### Bug 1: LOCAL_RUNTIME_PATHS never copied

`qq-worktree.py` has two distinct worktree-creation paths:

| Path | Trigger | What gets copied |
|---|---|---|
| **A. `command_create`** (CLI: `qq-worktree.py create --name X`) | Agent fallback when `EnterWorktree` unavailable | `git worktree add` + `copy_local_runtime_files()` + `copy_baseline_state_files()` + `ensure_runtime_cache_seed()` + writes `.qq/state/worktree.json` |
| **B. `EnterWorktree` tool** (Claude Code built-in) | Default path in `/qq:execute` and `/qq:go` skills since v1.16.21 | Only `git worktree add`. **No qq-worktree.py code runs.** |

`LOCAL_RUNTIME_PATHS = [".mcp.json", ".claude/settings.local.json", "qq.yaml", "AGENTS.md", "CLAUDE.md", "scripts"]` are copied **only on path A**. On path B these files exist in the source worktree but are **absent in the new worktree** (unless they're tracked by git, in which case `git worktree add` brings them).

In a typical Unity project that uses qq:
- `scripts/` is **synced from the qq plugin** by `auto-sync.sh` at SessionStart and is **not git-tracked** (it's in `.gitignore` as `scripts/__pycache__/` etc, but the `scripts/` directory itself is project-level cruft for some teams).
- Even if `scripts/` IS tracked, `auto-sync.sh` rewrites parts of it on SessionStart, so the worktree's copy goes stale immediately if it was just `git worktree add`'d.
- `.mcp.json` is typically generated by qq install and not always tracked.
- `qq.yaml` is project config — usually tracked, so this one is fine.
- `AGENTS.md` / `CLAUDE.md` — usually tracked, also fine.

**The agent's complaint** was specifically about `scripts/platform/` missing — that's `scripts/platform/detect.sh` etc, the platform-detection layer. If `scripts/` wasn't copied, every qq script invocation in the new worktree fails because they all `source scripts/platform/detect.sh` at the top.

**Confirmed behavior of `copy_runtime_path()`** (line 432-438):
```python
def copy_runtime_path(src: Path, dst: Path) -> None:
    if src.is_dir():
        dst.mkdir(parents=True, exist_ok=True)
        shutil.copytree(src, dst, dirs_exist_ok=True)
        return
    dst.parent.mkdir(parents=True, exist_ok=True)
    shutil.copy2(src, dst)
```
`shutil.copytree` is recursive, so when path A runs, it correctly copies `scripts/` including `scripts/platform/`. The function is correct. The bug is that **path B never calls this function**.

### Bug 2: Library seed is a real byte copy on Windows

`clone_copy_tree()` (line 500-525) selects strategy:
1. **macOS**: `cp -cR` → APFS clonefile (COW, instant)
2. **Linux/Windows with rsync on PATH**: `rsync -a --delete` → real I/O
3. **Fallback**: `shutil.copytree(dirs_exist_ok=True)` → real Python copy

Windows users typically run Git for Windows' bash, which **does not bundle rsync**. The fallback `shutil.copytree` performs a full byte copy of `Library/`, which for a Unity project is 1–5 GB. On a fast SSD this takes 30 seconds to 2 minutes per worktree creation.

The agent's "49s baseline compile" reported in the original incident is likely **Library copy ~30s + cold compile ~20s**, not pure compile time.

**NTFS supports hard links** (`mklink /H`, Python `os.link`). Hard-linking each file in `Library/` instead of byte-copying would reduce the seed time from 30–120 s to **<1 s**. This is identical to how macOS clonefile works conceptually (both create a new dirent pointing at the existing inode/file content), except hard links don't track divergent writes (they're aliases, not COW). For `Library/` this is fine because Unity treats it as a regenerable cache — if anything writes to it from one worktree, the other is OK to be invalidated.

## Proposed solution

### Fix 1: New `seed-local-runtime` subcommand

Add a new `qq-worktree.py` subcommand that runs only the `LOCAL_RUNTIME_PATHS` copy logic, intended to be called immediately after `EnterWorktree`.

**Signature**:
```bash
qq-worktree.py seed-local-runtime --project <new-worktree-path> --source <source-worktree-path>
```

**Behavior**:
- Resolves `--project` and `--source` to absolute paths
- Verifies both are valid git worktrees of the **same underlying repo** by comparing `git rev-parse --path-format=absolute --git-common-dir` (NOT `--show-toplevel`, which differs by design between linked worktrees)
- Runs `copy_local_runtime_files(source, project)` (existing, unchanged)
- Runs `copy_baseline_state_files(source, project)` (existing, unchanged)
- Runs `copy_baseline_run_records(source, project)` (existing, unchanged) — REVISED, was missing in Round 1 draft
- **Writes `.qq/state/worktree.json` metadata** with `managedBy=qq`, `sourceWorktreePath`, `sourceBranch`, etc. — REVISED, was open question in Round 1 draft. This ensures `EnterWorktree` worktrees behave identically to `command_create` worktrees from the perspective of `build_status` / `commit-push` Type B closeout / `qq-codex-exec.py --add-dir` injection / `qq-project-state.py` / `qq-doctor.py`.
- Returns JSON with `{ok, copied_local, copied_baseline_state, copied_baseline_run_records, metadata_path, ...}`
- Exits 0 on success, 1 on error

**Implementation outline** (~50 lines of new code):
```python
def command_seed_local_runtime(args: argparse.Namespace) -> dict[str, Any]:
    project_dir = Path(args.project).resolve()
    source_dir = Path(args.source).resolve()
    if not project_dir.is_dir():
        raise RuntimeError(f"Project dir does not exist: {project_dir}")
    if not source_dir.is_dir():
        raise RuntimeError(f"Source dir does not exist: {source_dir}")
    if project_dir == source_dir:
        raise RuntimeError("Project and source must be different directories")

    # Verify both are linked worktrees of the same underlying repo by
    # comparing git common dirs (NOT toplevels — linked worktrees have
    # different toplevels by design).
    src_common = run_git(source_dir, "rev-parse", "--path-format=absolute", "--git-common-dir").stdout.strip()
    dst_common = run_git(project_dir, "rev-parse", "--path-format=absolute", "--git-common-dir").stdout.strip()
    if Path(src_common).resolve() != Path(dst_common).resolve():
        raise RuntimeError(
            f"Source and project are not linked worktrees of the same repo "
            f"(source common-dir: {src_common}, project common-dir: {dst_common})"
        )

    # ALSO verify project_dir is a registered worktree root, not just any
    # subdirectory inside the source repo. `--git-common-dir` matching is
    # necessary but not sufficient — a regular subdir of source_dir also
    # passes that check. Without this, seed-local-runtime would copy files
    # to a non-worktree subdirectory and write .qq/state/worktree.json there,
    # making the metadata invisible to downstream consumers.
    worktree_list = run_git(source_dir, "worktree", "list", "--porcelain").stdout
    registered_paths = set()
    for line in worktree_list.splitlines():
        if line.startswith("worktree "):
            registered_paths.add(Path(line[len("worktree "):]).resolve())
    if project_dir.resolve() not in registered_paths:
        raise RuntimeError(
            f"--project {project_dir} is not a registered git worktree of the source repo. "
            f"Use `git worktree add` (or `qq-worktree.py create`) to create the worktree first, "
            f"then run seed-local-runtime."
        )

    copied_local = copy_local_runtime_files(source_dir, project_dir)
    copied_state = copy_baseline_state_files(source_dir, project_dir)
    copied_run_records = copy_baseline_run_records(source_dir, project_dir)

    # Write .qq/state/worktree.json so downstream consumers
    # (commit-push, qq-codex-exec, qq-project-state, qq-doctor, build_status)
    # treat this worktree as managed.
    branch = current_branch(project_dir)
    metadata = {
        "managedBy": MANAGED_BY,
        "metadataVersion": METADATA_VERSION,
        "createdAt": utc_now_iso(),
        "engine": resolve_project_engine(project_dir),
        "worktreeName": project_dir.name,
        "branch": branch,
        "sourceBranch": current_branch(source_dir),
        "sourceWorktreePath": str(source_dir),
        "currentPath": str(project_dir),
        "copiedLocalRuntimeFiles": copied_local,
        "copiedBaselineStateFiles": copied_state,
        "copiedBaselineRunRecords": copied_run_records,
        "createdVia": "seed-local-runtime",  # distinguish from command_create
    }
    metadata_file = write_metadata(project_dir, metadata)

    return {
        "ok": True,
        "action": "seed-local-runtime",
        "projectDir": str(project_dir),
        "sourceDir": str(source_dir),
        "copiedLocalRuntimeFiles": copied_local,
        "copiedBaselineStateFiles": copied_state,
        "copiedBaselineRunRecords": copied_run_records,
        "metadataPath": str(metadata_file),
    }
```

### Fix 3: Windows hard-link path in `clone_copy_tree()` — UNITY ONLY

**REVISED in Round 1**: hardlink is **scoped strictly to Unity** because Godot's `.godot/uid_cache.bin` and Unreal's `Intermediate/Build/*.modules` are rewritten in place by their respective toolchains. Hard linking those would corrupt the source worktree's cache when either worktree compiles. Unity's import pipeline writes Library artifacts via atomic `new file → rename`, making it the only currently-supported engine where aliasing is safe.

Add a Windows-specific strategy before the `shutil.copytree` fallback, gated on engine:

```python
def clone_copy_tree(source_dir: Path, target_dir: Path, *, allow_hardlink: bool = False) -> str:
    target_dir.mkdir(parents=True, exist_ok=True)
    if sys.platform == "darwin":
        # ... existing clonefile path
    rsync_binary = shutil.which("rsync")
    if rsync_binary:
        # ... existing rsync path

    # NEW: hardlink path — only when caller explicitly opts in.
    # Caller must guarantee the cache directory is safe to alias
    # (i.e., the engine writes files atomically, never in-place).
    # As of 2026-04-07: only Unity Library is known-safe.
    #
    # CRITICAL SAFETY: stage hardlinks into a sibling temp dir, atomic-rename
    # on full success, rmtree the staging dir on ANY failure. We never leave
    # partially hard-linked files in target_dir, because if we fell through to
    # shutil.copytree on a partially-linked target, copytree's O_TRUNC on
    # existing dst files would write through the hard link and corrupt the
    # source's cache. The staging approach makes hardlink success/failure
    # all-or-nothing.
    if allow_hardlink and (sys.platform == "win32" or sys.platform.startswith("linux")):
        staging = target_dir.parent / (target_dir.name + ".hardlink-staging")
        if staging.exists():
            shutil.rmtree(staging, ignore_errors=True)
        try:
            _hardlink_tree(source_dir, staging)
        except (OSError, NotImplementedError):
            shutil.rmtree(staging, ignore_errors=True)
            # Fall through to copytree on a clean (empty) target
        else:
            # All hardlinks succeeded — atomic rename onto target
            if target_dir.exists():
                shutil.rmtree(target_dir)
            staging.rename(target_dir)
            return "hardlink"

    shutil.copytree(source_dir, target_dir, dirs_exist_ok=True)
    return "copytree"


def _hardlink_tree(src: Path, dst: Path) -> None:
    """Create dst tree mirroring src, using hard links for files. Caller is
    responsible for ensuring dst is empty (use a staging dir) so that partial
    failures don't leave hard-linked files in the final target path."""
    for src_path in src.rglob("*"):
        rel = src_path.relative_to(src)
        dst_path = dst / rel
        if src_path.is_dir():
            dst_path.mkdir(parents=True, exist_ok=True)
        else:
            dst_path.parent.mkdir(parents=True, exist_ok=True)
            os.link(src_path, dst_path)
```

**Why staging instead of in-place + rmtree-on-failure**:
- Staging is atomic from the caller's perspective — `target_dir` only ever contains a complete hardlink tree or nothing.
- If we hardlinked directly into `target_dir` and then rmtreed on failure, there's a race window where a concurrent reader could see partial state. With staging, the rename is the only observable transition.
- `staging.rename(target_dir)` is atomic on POSIX and on Windows when both paths are on the same filesystem (NTFS supports this via `MoveFileEx` with `MOVEFILE_REPLACE_EXISTING`).
- Failure case is also clean: `rmtree(staging)` only removes the staging dir, never touches `target_dir` or `source_dir`.

And update `ensure_runtime_cache_seed()` to pass the flag based on engine:

```python
def ensure_runtime_cache_seed(project_dir: Path, source_worktree: Path | None, *, refresh: bool = False) -> RuntimeCacheSeedResult:
    cache_dir, source_cache, target_cache = runtime_cache_paths(project_dir, source_worktree)
    # ... existing checks
    engine = resolve_project_engine(project_dir)
    allow_hardlink = (engine == "unity")  # Unity Library is the only known-safe engine cache for aliasing
    try:
        strategy = clone_copy_tree(source_cache, target_cache, allow_hardlink=allow_hardlink)
    except Exception as exc:
        # ... existing
```

**Why hard link vs symlink**:
- Symlinks on Windows require admin rights or developer mode → unreliable
- Hard links work without elevated permissions
- Hard links cannot cross filesystems → fall through to copytree if needed

**Side effect concerns** (Unity only):
- Concurrent writes from one worktree are visible in the other (no COW). For Unity `Library/` this is acceptable because Unity rebuilds on demand and atomic-renames artifact files.
- **Risk**: if a user opens BOTH worktrees in Unity simultaneously, hard-linked `Library/` could cause one Editor to clobber the other's import state. Document in `docs/en/worktrees.md` and `docs/zh-CN/worktrees.md`. The "one editor per worktree" rule is already in best practices.
- **Why NOT extend to Godot/Unreal**: Godot's `uid_cache.bin` and editor state files, and Unreal's UBT-managed `.modules` / `Makefile.bin` are rewritten in place during normal operation. Hard-linking would corrupt the source worktree's cache as soon as either worktree compiles. This is verified by reading Codex finding 4 verification (subagent confirmed Unity is the only safe engine).

## Alternatives considered

1. **Make `EnterWorktree` itself run qq-worktree.py copy logic**: Rejected. `EnterWorktree` is Claude Code built-in, can't modify.
2. **Make `seed-runtime-cache` do BOTH Library AND local runtime files**: Rejected. Violates single-responsibility — `seed-runtime-cache` is named for the Library cache. Mixing in `LOCAL_RUNTIME_PATHS` makes the command name lie. Better to add a separate command.
3. **Make `qq-worktree.py create` the only path** (deprecate `EnterWorktree`): Rejected. EnterWorktree gives auto-CWD-switch which is a real UX win; manually using qq-worktree.py requires the user to reopen Claude Code in the new path.
4. **Use symlinks instead of hard links on Windows**: Rejected. Symlinks need elevated permissions.
5. **Use Windows volume shadow copy / reflink**: Rejected. Complex, requires admin, not portable.
6. **Defer Library seed entirely (let user wait for cold import)**: Rejected. Cold import in Unity is 30-60s vs 1-2s for hardlink seed — same order of magnitude as the current bug. Hardlink wins.

## Implementation Notes (deferred from Round 2 review)

### Codex Round 2 Suggestion: skill cache step doesn't persist `runtimeCacheSeed` metadata

`command_seed_runtime_cache` (qq-worktree.py:866 area) only writes the `runtimeCacheSeed` block of `.qq/state/worktree.json` when `--source` is omitted (because it expects to read source from existing metadata). The skill currently calls it with `--source <SOURCE_PROJECT>`, so even after a successful cache seed, `qq-project-state` and `qq-doctor` will report blank `worktreeRuntimeCacheSeedState`.

**Resolution at implementation time**: after `seed-local-runtime` writes the metadata (including `sourceWorktreePath`), the SKILL.md can call `seed-runtime-cache` **without `--source`** because the metadata is already there. So:

```bash
# Before (current Round 1 SKILL flow, has Round 2 suggestion bug)
${CLAUDE_PLUGIN_ROOT}/bin/qq-worktree.py seed-runtime-cache --project . --source "<SOURCE_PROJECT>"

# After Round 2 fix (call seed-local-runtime first, then bare seed-runtime-cache)
${CLAUDE_PLUGIN_ROOT}/bin/qq-worktree.py seed-local-runtime --project . --source "<SOURCE_PROJECT>"
${CLAUDE_PLUGIN_ROOT}/bin/qq-worktree.py seed-runtime-cache --project .
```

This way `command_seed_runtime_cache` reads `sourceWorktreePath` from the freshly-written metadata, and persists the seed result back into the same file.
